## 8. 문단 자료 구조

스펙 4.1 ~ 4.4. **가장 중요한 섹션.**

### 8.1. 문단 리스트 구조

```
문단 → 문단 → ... → 빈 문단(종료 마커)
```

**종료 조건 (스펙 확인)**: `char_count == 0`이면 리스트 끝.
- `lc(줄 수)`는 종료 판단에 사용하지 않는다.

### 8.2. 문단 정보 (가변 43 또는 230 bytes)

| 오프셋 | 크기 | 필드 | 설명 |
|--------|------|------|------|
| 0 | byte | follow_prev_shape | 0=새 문단 모양 저장됨, 이외=앞 따라감 |
| 1 | word | char_count | **0=빈 문단(리스트 끝)**, 이외=글자 수 (hchar 단위) |
| 3 | word | line_count | 문단의 줄 수 |
| 5 | byte | has_char_shape | 0=대표 모양만, 이외=각 글자 모양 별도 저장 |
| 6 | byte | extra_flags | bit0=단나눔, bit1=페이지나눔, bit2=조판나눔, ... |
| 7 | dword | special_char_flags | 특수 문자 존재 비트맵 |
| 11 | byte | style_index | 스타일 인덱스 |
| 12 | 31B | repr_char_shape | 대표 글자 모양 |
| 43 | 187B | para_shape | **follow_prev_shape == 0일 때만 저장** |

전체 크기: `follow_prev_shape == 0` → 230B, 이외 → 43B

**주의**: `char_count == 0`이면 빈 문단(리스트 종료 마커)이며, `follow_prev_shape`와 무관하게 줄 정보/글자 모양/글자를 읽지 않는다.

> **실측**: `para#13 @ 0xE962 (fp=0, cc=0, lc=0)` → para_shape 187B 소비 후 `0xEA48`에 도달. 빈 문단도 `follow_prev=0`이면 para_shape 187B가 저장되어 있다.

### 8.3. 줄 정보 (14 bytes × line_count)

| 오프셋 | 크기 | 필드 | 설명 |
|--------|------|------|------|
| 0 | word | start_pos | 줄 시작 위치 (hchar 단위) |
| 2 | hunit | space_adjust | 공백 보정값 |
| 4 | hunit | line_height | 줄 높이 |
| 6 | byte[6] | reserved | 예약 |
| 12 | word | page_flags | 단/페이지 구분 플래그 |

`LINE_INFO_SIZE = 14`

### 8.4. 글자 모양 정보

`has_char_shape != 0 AND char_count > 0`일 때만 존재.

```
{ byte flag, [CharShape 31B] } × char_count
flag == 1 → 앞 글자 모양 따라감 (CharShape 없음)
flag != 1 → 새 글자 모양 (CharShape 31B 뒤따름)
```

`CHAR_SHAPE_SIZE = 31`

### 8.5. 글자들 (hchar 스트림)

`char_count`개의 hchar(2바이트)가 이어짐.

**hchar 인코딩**:
- `code > 31`: 일반 문자 (JOHAB 2바이트 빅엔디언)
  - `code < 0x80`: `chr(code)` (ASCII)
  - `code >= 0x80 AND 상위바이트 >= 0x84`: `struct.pack(">H", code).decode("johab", errors="replace")`
  - **유효 Johab 범위**: 상위바이트 `0x84~0xD3` (첫 글자 `0x8441`). `0x84` 미만이면 렌더링 캐시 데이터 → 빈 문자열 반환
- `code == 13`: 문단 끝 (CR, `\n` 출력)
- `code == 9`: 탭 (`\t`)
- `code <= 31`: 특수 제어 코드 (9장 참조)

### 8.6. 렌더링 캐시 para 판별 기준 (실측)

문단 파싱 중 다음 조건이면 렌더링 캐시 데이터로 간주하고 즉시 중단:

| 조건 | 설명 |
|------|------|
| `char_count > 300` | 비현실적으로 큰 글자 수 |
| `char_count > 0 AND line_count == 0` | 정상 문단은 반드시 `line_count >= 1` |

**실측 근거**:
- 정상 문단 최대 `char_count` = **173** (`test97.hwp` 기준)
- 렌더링 캐시 para: `char_count=256, line_count=0` 패턴 반복 발견
- `line_count == 0` 조건이 가장 명확한 판별자

```python
# _parse_paragraph_list 내 구현
if para.info.char_count > _MAX_CC:          # _MAX_CC = 300
    stream.seek(pos_before)
    break
if para.info.char_count > 0 and para.info.line_count == 0:
    stream.seek(pos_before)
    break
```

## 12. 렌더링 캐시 (ExtraBlock)

### 12.1. 구조

`body_raw[0xEA50]` 이후 약 5.6 MB.

- **7페이지 × 256,000 bytes** = 1,792,000 bytes의 렌더링 캐시 데이터
- 렌더링 캐시 = 한글 97이 화면 출력용으로 저장하는 페이지 비트맵/텍스트 캐시
- **원본 문단 데이터와 다름**: 폰트 렌더링 코드가 섞여 있어 기호 등이 변환됨

### 12.2. 렌더링 캐시 내 텍스트 레코드 형식

렌더링 캐시 안에는 문단별 텍스트가 레코드 형식으로 저장되어 있다.

**Type 1 레코드**:
```
[sep×n≥4][null×nc][flag×fc≥1][text(JOHAB hchar)][0x0D 0x00]
```

**Type 2 레코드** (double-header, nc=10, fc=0일 때):
```
[sep×n≥4][null×10][0x0002][null×nc2][flag×fc2][text][0x0D 0x00]
```

`sep_bytes = {0x64, 0xFF, 0xFE}`

### 13.2. 수정된 버그 이력

#### 버그 A: `Paragraph.is_empty` 종료 조건 오류 (수정 완료)

**증상**: `char_count==0, line_count==1`인 종료 마커를 빈 문단으로 인식하지 못함 → 2페이지 이후 본문 미파싱.

**원인**: `is_empty` 조건에 `line_count == 0`을 AND 조건으로 포함.

**수정**: `char_count == 0`만으로 판단.

```python
@property
def is_empty(self) -> bool:
    return self.info.char_count == 0
```

#### 버그 B: `HChar.to_string` — 유효하지 않은 Johab 코드 처리 (수정 완료)

**증상**: `code=0x0600` 등 렌더링 캐시 데이터가 Johab 디코딩을 통과해 깨진 문자 출력.

**원인**: Johab 코덱이 `b'\x06\x00'`을 `'\x06\x00'`으로 디코딩 성공(오류 없음) → 제어문자 통과.

**수정**: 상위바이트 `< 0x84`이면 유효하지 않은 Johab으로 간주해 즉시 `""` 반환.

```python
high_byte = (self.code >> 8) & 0xFF
if high_byte < 0x84:
    return ""  # 유효 Johab 범위 0x8441~ 미만
```

#### 버그 C: `_parse_paragraph_list` — 렌더링 캐시 para 누출 (수정 완료)

**증상**: `char_count=256, line_count=0`인 렌더링 캐시 para가 `_MAX_CC=300` 체크를 통과해 깨진 텍스트 6개 수집.

**발생 경로**: `_parse_box_after_ident(버튼, box_type=3)` → inner para list + 캡션 para list 파싱 종료 → 외부 `_parse_paragraph_list` while 루프가 계속 실행 → 뒤따르는 렌더링 캐시 para(`cc=256, lc=0`) 읽기.

**수정**: `line_count == 0` 조건 추가.

```python
if para.info.char_count > _MAX_CC:
    stream.seek(pos_before)
    break
if para.info.char_count > 0 and para.info.line_count == 0:
    stream.seek(pos_before)
    break
```

#### 버그 D: `Paragraph.read_from_stream` — line_count 대량 바이트 소비 (수정 완료)

**증상**: offset 467,875 부근에서 `cc=3, lc=53248` garbage para 파싱 → `14 × 53248 = 745,472 bytes` 소비 → `current_end`가 1.2M으로 점프 → '붙임', '확약서', '보안서약' 등 이후 전 섹션 미추출.

**원인**: `Paragraph.read_from_stream`에 `line_count` 상한 검사 없음 → 렌더링 캐시의 garbage `lc` 값(`53248`)을 그대로 사용해 `14 × 53248`바이트 읽기 시도.

**수정**: `line_count > 200` → `stream.seek(pos_start)` + `return None`.

```python
_MAX_LC = 200

pos_start = stream.tell()
info = ParaInfo.from_stream(stream)
if info.line_count > _MAX_LC:
    stream.seek(pos_start)
    return None
```

**효과**: to_md 19,824자 → 43,832자 (2.2배 증가).

#### 버그 E: `_parse_box_after_ident` — 캡션 렌더링 캐시 오파싱 (수정 완료)

**증상**: 표 파싱 후 캡션 위치에서 `fp=21, cc=256, lc=1024` (렌더링 캐시)를 유효한 para list로 오인식 → 다음 섹션의 본문 전체 소비.

**원인**: 캡션 para list 시작 전 헤더 유효성 검증 없이 바로 `_parse_paragraph_list` 호출.

**수정**: 6바이트 peek으로 para info 헤더 유효성 먼저 검증.

```python
cap_peek = stream.read(6)
if len(cap_peek) == 6:
    fp_byte = cap_peek[0]
    cc = struct.unpack_from('<H', cap_peek, 1)[0]
    lc = struct.unpack_from('<H', cap_peek, 3)[0]
    hcs = cap_peek[5]
    valid_cap = (fp_byte in (0, 1) and cc <= _MAX_CC and lc <= 200 and hcs <= 1)
    if valid_cap:
        stream.seek(-6, 1)
        _parse_paragraph_list(stream)
```

#### 버그 F: `_parse_paragraph_list` — 코드 10 처리 후 while 미종료 (수정 완료)

**증상**: 표 포함 문단(코드 10) 처리 후 같은 para list의 다음 단락까지 소비 → 동일 텍스트 중복 수집.

**원인**: `_parse_box_after_ident` 호출 후 `break` 없이 while 루프 계속 실행.

**수정**: `should_end_list = True` 플래그 → while break.

```python
should_end_list = False
for hc in para.chars or []:
    if hc.char_type.value == "special":
        if hc.code == 10:
            parsed_table = _parse_box_after_ident(stream)
            if parsed_table is not None:
                paragraphs.append(parsed_table)
            should_end_list = True
        break
if should_end_list:
    break
```

#### 버그 G: `_is_valid_text` — CharShape.attr=255 garbage 출력 (수정 완료)

**증상**: `黜` 등 렌더링 캐시 garbage 텍스트가 `_is_valid_text` 필터를 통과해 출력.

**원인**: `CharShape.attr == 255`인 문단은 렌더링 캐시에서 비롯된 garbage임에도 필터 기준 없음.

**수정**: `char_shape` 파라미터 추가, `attr == 255` 조건 추가.

```python
def _is_valid_text(text: str, char_shape: Optional[Any] = None) -> bool:
    if "\ufffd" in text:
        return False
    if len(text) > 3 and text.count("\n") / len(text) > 0.5:
        return False
    if char_shape is not None and getattr(char_shape, "attr", 0) == 255:
        return False
    return True
```

#### Fix H: `ParsedTable.to_text()` 미구현 (수정 완료)

**증상**: `to_txt`에서 표 내용이 전혀 출력되지 않음.

**원인**: `ParsedTable`에 `to_text()` 메서드 없음 → `getattr(elem, "to_text", None)` → None 반환.

**수정**: `to_text()` 및 `to_markdown()` 메서드 추가 (`parsed_elements.py`).

```python
def to_text(self) -> str:
    """표/텍스트박스 셀 내용을 탭 구분 텍스트로 변환."""
    ct = self.cell_texts
    if not ct:
        return ""
    return "\n".join("\t".join(row) for row in ct)

def to_markdown(self) -> str:
    """표를 마크다운 표 형식으로 변환."""
    ct = self.cell_texts
    if not ct:
        return ""
    cols = max(len(r) for r in ct)
    lines = []
    for row_idx, row in enumerate(ct):
        padded = row + [""] * (cols - len(row))
        cells = [c.replace("\n", " ").replace("|", "\\|").strip() for c in padded]
        lines.append("| " + " | ".join(cells) + " |")
        if row_idx == 0:
            lines.append("| " + " | ".join(["---"] * cols) + " |")
    return "\n".join(lines)
```

## 14. 전체 body 파싱 전략

HWP 97 body는 root para list 1개 + 이후 분산된 para list들로 구성된다.
root para list만 파싱하면 전체 문서의 일부(표지 등)만 추출된다.

### 14.1. 구조 (실측)

| 항목 | 값 |
|------|-----|
| body_bytes 크기 | 5,665,450 bytes (글꼴+스타일 소비 후) |
| root para list 종료 | offset 4,593 |
| total iteration | 285 (skip 219회 포함) |
| 최종 current_end | 5,665,450 (body 전체 순회 완료) |
| MAX_ITER 초과 | **없음** |

### 14.2. 파싱 전략: `_find_next_para_list_start` 기반 순차 탐색

실제 구현은 고정 마커(`00 05 00 01`) 방식이 아닌 **para info 헤더 조건 기반 스캔**이다.

```python
def _find_next_para_list_start(body: bytes, start: int, max_scan: int = 20000) -> int:
    # body[start:start+max_scan] 범위에서 유효한 para info 헤더 탐색
    # 조건: fp in {0,1}, cc in [1,300], lc in [1,200], hcs in {0,1}
    ...

def _parse_all_body_lists(body_stream: BinaryIO) -> List:
    body_bytes = body_stream.read()
    all_paragraphs: List = []
    current_end = 0
    iteration = 0
    MAX_ITER = 2000

    # 1. root para list 파싱 + _is_valid_text 필터
    root_stream = io.BytesIO(body_bytes)
    root_paras = _parse_paragraph_list(root_stream)
    for para in root_paras:
        if isinstance(para, ParsedParagraph):
            if _is_valid_text(para.text, para.char_shape):
                all_paragraphs.append(para)
        else:
            all_paragraphs.append(para)
    current_end = root_stream.tell()

    # 2. 순차 스캔: 20KB 범위에서 다음 para list 시작점 탐색
    while iteration < MAX_ITER and current_end < len(body_bytes) - 43:
        next_start = _find_next_para_list_start(body_bytes, current_end, max_scan=20000)
        if next_start < 0:
            current_end += 20000
            iteration += 1
            continue
        seg_stream = io.BytesIO(body_bytes[next_start:])
        seg_paras = _parse_paragraph_list(seg_stream)
        seg_end = seg_stream.tell()
        if seg_end <= 0 or next_start + seg_end <= current_end:
            current_end = next_start + 1
            iteration += 1
            continue
        for para in seg_paras:
            if isinstance(para, ParsedParagraph):
                if _is_valid_text(para.text, para.char_shape):
                    all_paragraphs.append(para)
            else:
                all_paragraphs.append(para)
        current_end = next_start + seg_end
        iteration += 1
    return all_paragraphs
```
